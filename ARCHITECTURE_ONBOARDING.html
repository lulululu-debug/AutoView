1. 这是个什么项目

双边 AI 面试基础设施：HR 上传 JD / 岗位要求 / 公司资料， 候选人面试前上传 Resume；系统结合 JD 与 Resume 自动生成面试计划、 执行多轮面试与追问、产出结构化评估报告。

HR 端职责

  • 创建职位、上传 JD/公司资料
  • 选择 track (校招 campus / 社招 lateral)
  • 查看候选人列表 + 状态 (plan_pending / ready / completed / reviewed)
  • 查看面试阶段视图 + 评估报告
  • 人工复核：留备注 + 维度覆盖 + 三态 decision

候选人端职责

  • 上传 Resume (PDF / docx 自动解析, 或 paste)
  • 等待 Planner 生成面试计划
  • 按 stage 序列完成 7-9 道题面试
  • 面试过程可中断恢复 (Redis 会话热存储)

面试 stage 序列 (Sprint 5.5 起)

Track序列题数
campus (校招) self_intro → knowledge ×3 → project ×2 (lazy) → scenario ×1 7
lateral (社招) self_intro → project ×3 (lazy) → scenario ×2 → knowledge ×1 7

lazy = plan 阶段只占位 text="", 进 project stage 时用候选人 intro_text + Resume RAG 现场生成。让题目真正反映候选人自我介绍里提到的内容。

6. 一次面试的完整数据流

[HR] POST /jobs (api/routes/jobs.py) ↓ db.save_job ↓ BG: ingestion.ingest_jd + ingest_company_material → Milvus documents [候选人] GET /interview/{job_id} ← 看 JD + 上传 Resume [候选人] POST /jobs/{id}/candidates/parse-resume ← Sprint 5.8 PDF/docx 解析 [候选人] POST /jobs/{id}/candidates {resume: text} ↓ db.save_candidate ↓ BG: ingestion.ingest_resume → Milvus documents ↓ BG: planner.plan(job, candidate) ├─ embed query → Milvus search_questions (knowledge / scenario) ├─ llm.complete (knowledge×N + scenario×N 精修) └─ project 题留 lazy 占位 (text="") ↓ db.save_plan + cache.save_plan [候选人] GET /jobs/{id}/candidates/{cid}/plan (轮询直到 200) [候选人] POST /interviews {candidate_id} ← start_session [候选人] POST /interviews/{sid}/answers {text} ← submit_answer 多次 (1) 落 CandidateAnswer 进 session.answers (2) 若是 self_intro → 落 session.intro_text (3) [ASSESSOR_ENABLED=true 时] assessor.assess → session.assessments (4) interviewer.next_turn → Question / FollowUp / None (5) 若 next 是 lazy 占位 → planner.resolve_lazy_questions ├─ embed query → Milvus search_documents (kind=resume) ├─ llm.complete (project_RAG_SYSTEM, intro_text + chunks) └─ cache.save_plan + db.save_plan ← Sprint 5.8 fix 同步 PG [候选人] POST /interviews/{sid}/finalize ← done 页 fire-and-forget ↓ analyzer.analyze (空占位) ↓ evaluator.evaluate ├─ compute_coverage (从 session.assessments + plan.competencies) ├─ embed candidate_excerpt → Milvus search_documents (kind=jd|cm) ├─ llm.complete (summary 生成) └─ insufficient_competencies → summary 前缀加"证据不充分" ↓ db.save_session + db.save_report ↓ cache.delete_session + delete_plan (从 Redis 清掉) [HR] GET /hr/jobs/{j}/candidates/{c} ← 看到状态变 completed [HR] GET /hr/sessions/{sid} ← 拉 session (含 assessments) [HR] GET /hr/reports/{report_id} ← 拉 report [HR] PATCH /hr/reports/{rid}/review ← 人工复核

10. 合规与硬约束

这些规则不是建议，破坏任意一条 = 合规事故。详情见 ARCHITECTURE.md §7。

  1. 分数与软信号分区: EvaluationReport.content_scores (内容维度，进总分) 与 performance_observations (表现维度，软信号) 严格分开; overall 只由 content_scores 加权得出。任何新增表现/软信号永远走 PerformanceObservation。
  2. 多模态软信号 (Sprint 7): 只能作为参考证据，绝不可作为自动淘汰的唯一依据; 必带置信度 + 可关闭 + 偏见审计。
  3. 不展示 LLM-as-judge 数字给 HR: AnswerAssessment.sufficiency / confidence 是中间产物，校准前不可信。HR UI 只展示自然语言字段 (missing_signals / strengths / concerns / followup_goal)。
  4. 候选人不接触自己的报告: 候选人端永不返 EvaluationReport / DimensionScore。
  5. 不做动态补题: 题库由 plan + lazy project gen 一次确定，coverage 不够让 HR 复核兜底，不能 LLM 在线生成新题 (公平性 + 可复现性双坍方)。
  6. Assessor 准入校准: ASSESSOR_ENABLED 默认 false; 校准 eval (20-30 条手标样本) 过 + 人工 review 真 LLM 路径 → 才在部署层翻 true。
  7. 默认需人工复核: EvaluationReport.needs_human_review = True 是默认; AI 不直接做录用决定。
  8. Agent 之间不直接互调: 全部走 Orchestrator 路由; 数据 + I/O 通过 schema 类型流转，不通过函数调用。
  9. 启发式 fallback 永远不删: Sprint 0 的 _needs_followup 是 Assessor 的兜底, 启发式 sufficiency 计算是 LLM 失败时的兜底; 双路径永远共存。

怎么看日志？

followup 怎么知道有没有触发？

数 session.history 里 interviewer turn 的 ref_id: 在 plan.questions 里 = 主问题, 不在里面 = 追问。 或看 evals/test_assessor_integration 里的 _decide_followup 单元测试就明白决策逻辑。

怎么改一个 prompt？

  1. 找到对应文件 (src/agents/{planner,assessor,evaluator,interviewer}/__init__.py 顶部的 SYSTEM 常量)
  2. 必须先有 eval: 改 prompt 前确认结构性 eval / 校准 eval 存在; 校准 eval 不存在的改 = 5.6 起就违反约定
  3. 改完跑 python -m unittest discover -s evals 全绿

怎么加一个 agent / 新 LLM 调用？

  1. 建 src/agents/{name}/__init__.py 暴露一个动词函数
  2. 只 import src.llm + src.schemas (不互调 agent, 不接触 DB/Cache)
  3. 必带 llm.is_stub() 判断 + 启发式 fallback (双路径硬约束)
  4. 必带 timeout, 不允许"LLM 挂了整条链路就挂"
  5. orchestrator 里加路由 (不让 agent 自己调 cache/db)

怎么加一个新字段到 schema？

  1. 改 src/schemas/__init__.py, 字段加默认值 (兼容老 JSON)
  2. 若要落 PG: 改对应 ORM 列加 server_default, 然后 ALTER TABLE ADD COLUMN IF NOT EXISTS (dev 期手动, Sprint 1 不引 Alembic)
  3. 改对应 repository save/load
  4. 结构性 eval 验"老 JSON 缺这字段能解析" + "默认值正确"

ASSESSOR_ENABLED 怎么翻 true？

  1. evals/test_assessor_calibration 全绿 (启发式排序 sufficient > insufficient, gap ≥ 0.2)
  2. 用真 OPENAI_API_KEY 跑 calibration 数据集, 人工 review 每条 prediction 合理
  3. PR review 通过 → .env 翻 ASSESSOR_ENABLED=true
  4. 即使 true, Assessor 自己 LLM 失败仍走启发式 fallback (双路径永远共存)

Bearer / Cookie 鉴权选哪个？

Sprint 5.8 起双路径并存：

  • 浏览器 (HR 端 Web): 走 httpOnly cookie + SameSite=Strict, JS 看不到 token
  • evals / 脚本 / curl: 仍可走 Authorization: Bearer
  • require_hr_user 先看 cookie 后看 Bearer (任一命中即可)
  • 5.9+ 待 cookie 稳定后可考虑退役 Bearer 路径
