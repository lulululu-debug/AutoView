# ARCHITECTURE — AI Interview Platform

## 1. 目标与定位

一个多 agent 的 AI 视频面试平台。HR 上传 JD / 岗位要求 / 公司资料后，系统：

1. 自动生成结构化面试计划（轮次、考察维度、题库）
2. 自动执行多轮面试并按回答动态追问
3. 自动产出结构化评估报告

终态是「招聘端 + 候选人端」双边 AI 面试基础设施。

**设计原则**：核心价值与最大风险都在 agent 编排（计划质量、追问逻辑、报告结构），
而非实时媒体层。因此先用纯文本把 agent 链路做扎实，媒体层与多模态作为输入/输出适配器后接。

---

## 2. 多 Agent 分解

```
                    ┌─────────────────┐
   JD/岗位/公司资料 → │  Planner Agent   │ → InterviewPlan
                    │  (规划)          │   (轮次、维度、题库)
                    └────────┬─────────┘
                             ▼
                    ┌─────────────────┐
   CandidateAnswer ⇄ │ Interviewer Agent│ → Question / FollowUp
                    │  (面试官/追问)    │   依据回答决定是否追问
                    └────────┬─────────┘
                             ▼
                    ┌─────────────────┐
                    │ Evaluator Agent  │ → EvaluationReport
                    │  (评估)          │   按维度打分 + 证据
                    └─────────────────┘

   扩展层（后接）:
   ┌──────────────────────────┐
   │ Multimodal Analyzer Agent │  转写文本 → 语言信号
   │ (语言/语气/眼神)           │  音频     → 语气/韵律信号
   │                          │  视频帧   → 表情/视线信号
   └──────────────────────────┘
        ↓ 仅作为「参考证据」喂给 Evaluator（见第 7 节合规约束）
```

所有 agent 之间不直接互相调用，统一由 Orchestrator 传递上下文。

### 2.1 Planner Agent（规划）
- **输入**：JD、岗位要求、公司资料（后期经 RAG 召回相关片段）
- **输出**：`InterviewPlan`——若干轮次，每轮包含考察维度与候选问题
- **职责**：把岗位要求映射成可考察的能力维度，并为每个维度生成题目

### 2.2 Interviewer Agent（面试官 / 追问）
- **输入**：当前 `InterviewPlan` + 已有对话历史 + 候选人最新回答
- **输出**：下一个 `Question`，或基于回答的 `FollowUp`，或本轮结束信号
- **职责**：系统的对话主循环。追问逻辑是核心——根据回答的完整度/深度
  决定是否深挖。先做规则 + LLM 判断的简单版，后续迭代。

### 2.3 Evaluator Agent（评估）
- **输入**：完整 `InterviewSession`（对话历史）+（扩展）多模态信号
- **输出**：`EvaluationReport`——按维度打分、给出证据、总体结论
- **职责**：把对话证据化、结构化。报告中「内容维度」与「表现维度」分开呈现。

### 2.4 Multimodal Analyzer Agent（多模态分析，扩展）
- **输入**：音视频流 / 帧 / 转写文本
- **输出**：语言、语气、视线等软信号，附置信度
- **职责**：产出*参考证据*，不参与自动打分。详见第 7 节。

### 2.5 Orchestrator（编排器）
- 管理 `InterviewSession` 生命周期与状态机
- 在各 agent 间传递上下文
- 负责持久化（Postgres）与会话态缓存（Redis）

## 3. 核心数据结构（src/schemas/，pydantic）

- `JobContext`——JD + 岗位要求 + 公司资料
- `Competency`——单个考察维度（名称、描述、权重）
- `Question`——题目（所属维度、文本、类型）
- `InterviewPlan`——轮次列表，每轮含若干 Competency 与 Question
- `CandidateAnswer`——候选人回答（文本，后期含音视频引用）
- `InterviewSession`——一次面试的完整状态与对话历史
- `Signal`——多模态软信号（类型、值、置信度、来源）
- `DimensionScore`——单维度评分（分数、证据、是否含软信号）
- `EvaluationReport`——内容维度评分 + 表现维度评分 + 总体结论

## 4. 技术栈

| 层 | 选型 | 引入阶段 |
|----|------|----------|
| Agent 核心 | Python 3.11+ | Sprint 0 |
| LLM SDK | anthropic | Sprint 0 |
| Agent 编排 | 自研轻量编排（保持可控）| Sprint 0 |
| 数据结构 | pydantic | Sprint 0 |
| 持久化 | Postgres | Sprint 1 |
| 会话态 / 缓存 / 队列 | **Redis** | Sprint 1 |
| API 层 | FastAPI | Sprint 2 |
| 向量检索 / RAG | **Milvus** | Sprint 3 |
| 前端 | Next.js + TypeScript | Sprint 4–5 |
| 实时媒体 | WebRTC + 流式 STT/TTS | Sprint 6 |
| 多模态分析 | 视觉 / 音频模型 | Sprint 7 |
| 依赖管理 | uv | Sprint 0 |

### 4.1 Redis 的角色
- **会话状态**：进行中的 `InterviewSession` 是强状态的，Redis 做热存储，Postgres 做持久归档
- **缓存**：JD 解析结果、题库召回结果、可复用的 LLM 输出
- **任务队列 broker**：多模态分析耗时且需异步，Redis 作为队列后端（配合 RQ/Celery）

### 4.2 Milvus 的角色
- **题库语义检索**：按考察维度召回最相关题目，避免每次都让 LLM 凭空生成
- **公司资料 / JD RAG**：对上传材料切片、向量化，面试与评估时召回相关片段
- **回答语义分析**：候选人回答与岗位要求的语义匹配度

## 5. 关键流程（端到端）

```
HR 上传 JD/资料
   → (RAG 切片入 Milvus)
   → Planner 生成 InterviewPlan（题目从 Milvus 召回 + LLM 补充）
   → 候选人进入面试（会话态存 Redis）
   → Interviewer 循环：提问 → 收答案 → 判断追问 → 下一题/下一轮
   → (扩展) 音视频流并行送 Analyzer，软信号异步入库
   → 面试结束，Session 归档 Postgres
   → Evaluator 综合对话 + 软信号 → EvaluationReport
   → HR 端查看报告（含人工复核）
```

---

## 6. 分阶段策略（与 sprint.md 对应）

骨架（Sprint 0）只做纯文本、单进程、四 agent 串通、写死输入跑通一次。
之后逐层加：持久化与会话态 → API → 向量检索/RAG → 前端 → 实时媒体 → 多模态。
每一层都建立在已跑通的上一层之上，不一次性铺开。

---

## 7. 合规与公平性（多模态评价的硬约束）

用 AI 根据语言、语气、眼神评价候选人，是本项目风险最高的部分，架构上必须从一开始约束：

**已知风险**
- **眼神 / 视线**：眼神交流标准有强烈文化差异；神经多样性人群（如自闭谱系）天然不符合典型模式。作为评分项极易构成系统性歧视。
- **语气 / 情感分析**：现有模型在不同种族、性别、口音上有已知偏差。
- **法规**：AI 用于招聘在多地受监管（如纽约市对自动化招聘工具的偏见审计要求、欧盟 AI Act 将招聘 AI 列为高风险）。具体规定须咨询法律专业意见，且规则在持续变化。

**架构层面的强制约束**
1. 多模态信号仅作为给人看的**参考证据**，不自动计入总分，绝不作为自动淘汰的唯一依据
2. 报告中**内容维度（回答质量）与表现维度（语气/视线）分开呈现**，标明哪些是软信号
3. 每个信号可**单独关闭**；对受保护特征敏感的项（如眼神）默认保守处理
4. 保留**人工复核**环节与完整**审计日志**

做对这一层，"可审计、可解释、不歧视" 反而是面向 B 端客户的差异化卖点。
