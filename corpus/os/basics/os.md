# 操作系统

## 进程与线程基础

### 进程与线程的概念

进程：为了更好地描述和控制程序的并发，实现操作系统的并发性和共享性，引入了进程的概念。操作系统为每一个进程分配一个专门的数据结构：进程控制块 PCB（常驻内存），保存运行期间进程的数据，PCB 是进程存在的唯一标志。

进程是操作系统中进行资源分配和调度的最小单位，是正在运行的程序的实例。每个进程都有自己独立的地址空间、数据栈以及其他用于跟踪进程执行的辅助数据（如文件描述符、寄存器状态等）。

线程：线程是"轻量级进程"，它是基本的 CPU 执行单元，也是程序执行流的最小单元，由线程 ID、程序寄存器、寄存器集合和堆栈组成（即 CPU 调度的基本单元）。线程控制块（Thread Control Block, TCB）保存运行期间线程的数据，TCB 是线程存在的唯一标志。

线程是 CPU 调度（程序执行）的最小单位，是进程中的一个执行流。一个进程可以包含多个线程，它们共享该进程的地址空间、文件描述符、堆等资源，但每个线程有独立的栈空间和寄存器状态。

### 进程与线程的区别

引入线程是为了提高程序的并发性能。相较于进程，线程更加轻量，切换开销更低，适合处理高并发、高频率的任务，如 Web 服务、爬虫、图像处理等。

| 对比点 | 进程 | 线程 |
| --- | --- | --- |
| 地址空间 | 互相独立 | 共享所属进程的地址空间 |
| 通信方式 | 开销较大（需 IPC） | 开销较小（可共享内存） |
| 调度开销 | 创建和切换开销较大 | 创建和切换更轻量 |
| 崩溃影响 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能影响整个进程 |

进程和线程的关系：

- 一个线程只能属于一个进程，而一个进程可以有多个线程，但至少有一个线程。
- 资源分配给进程，同一进程的所有线程共享该进程的所有资源。
- 线程在执行过程中，需要协作同步。不同进程的线程间要利用消息通信的办法实现同步。
- 处理机分给线程，即真正在处理机上运行的是线程。
- 线程是进程内的一个执行单元，也是进程内的可调度实体。

区别的五个方面：

- 调度：进程是资源分配和调度的一个独立单元；而线程是 CPU 调度的基本单元。
- 并发性：不仅进程之间可以并发执行，同一个进程的多个线程之间也可以并发执行。线程并发执行过程中要进行同步和互斥，因为它们共享同一进程的所有资源。
- 拥有资源：进程是拥有资源的一个独立单位，线程不拥有系统资源，但可以访问隶属于进程的资源，并且线程共享整个进程的资源（寄存器、堆栈、上下文）。
- 系统开销：线程是轻量级的进程，它的创建和销毁所需要的时间比进程小很多，进程在切换和创建销毁时耗费的资源较大，系统开销较大。
- 健全性：进程有独立的地址空间，进程崩溃后，在保护模式下不会对其他的进程产生影响；线程有自己的堆栈和局部变量，但没有单独的地址空间，一个线程死掉就等于整个进程死掉，所以多进程的程序要比多线程的程序健壮。
- 创建和销毁：进程的创建调用 fork 或者 vfork，而线程的创建调用 pthread_create；进程结束后它拥有的所有线程都将销毁，而线程的结束不会影响同个进程中的其他线程的结束。

### 协程及其与线程的区别

协程是一种用户态的轻量级线程，是由用户控制的执行单元，拥有独立的执行上下文和栈。它的调度和上下文切换不依赖操作系统内核，而是由程序显式控制切换，可以在函数中间挂起和恢复执行，非常适合处理 I/O 密集型任务。

协程的切换完全在用户态完成，性能远优于线程，非常适合大量 I/O 场景（如网络请求、数据库操作），可以极大地减少上下文切换开销。

| 对比点 | 协程 | 线程 |
| --- | --- | --- |
| 所在空间 | 用户态 | 内核态 |
| 切换成本 | 极低（无需系统调用） | 较高（涉及内核态切换） |
| 并发模式 | 非并行（需配合事件驱动） | 可并行（依赖多核） |
| 使用场景 | 高并发 I/O、异步任务 | CPU 密集、并行处理 |

补充说明：协程本身不具备并行能力，但可以通过线程调度多个协程并发运行；线程的调度是由内核完成，涉及从用户态到内核态的切换。

## 进程状态与调度

### 进程状态与转换

操作系统中的进程通常具有以下几种状态：

- 就绪（Ready）：进程已经获得除 CPU 之外的所有资源，等待 CPU 分配。
- 运行（Running）：进程正在使用 CPU 执行。
- 阻塞（Blocked / Waiting）：进程正在等待某个事件（如 I/O）完成，无法继续执行。
- 新建（New）：进程正在创建中。
- 终止（Terminated / Exit）：进程执行完毕或被中止。
- 有些系统还有挂起状态（如"就绪挂起"、"阻塞挂起"），用于内存调度。

进程在就绪、运行、阻塞三种常见状态之间的转移机制：

- 运行态 → 阻塞态：进程因等待 I/O 或锁等资源主动放弃 CPU（主动行为）。
- 阻塞态 → 就绪态：等待的资源就绪，操作系统唤醒进程（被动行为）。
- 运行态 → 就绪态：时间片用完或被调度器抢占，进程让出 CPU（被动行为）。
- 就绪态 → 运行态：操作系统调度器分配 CPU 给该进程（被动行为）。

### 调度分级与调度方式

调度分级：

| 调度类型 | 又称 | 说明 |
| --- | --- | --- |
| 作业调度 | 高级调度 | 从外存中的后备作业中选择作业，分配资源并建立进程，使其获得竞争处理机的权利 |
| 内存调度 | 中级调度 | 选择挂起状态的进程重新调入内存（即从外存加载到内存） |
| 进程调度 | 低级调度 | 从就绪队列中选择一个进程，将处理机分配给它运行 |

发生频率比较：高级调度 < 中级调度 < 低级调度。

调度方式：

| 调度方式 | 适用场景 | 特点说明 |
| --- | --- | --- |
| 非抢占调度方式 | 批处理系统 | 当前进程运行完前不会被剥夺处理机 |
| 抢占调度方式 | 分时系统、实时系统等 | 当前进程运行中可能会被操作系统剥夺处理机 |

### 常见调度算法

操作系统使用调度算法决定哪个进程/线程优先获得 CPU 时间：

| 调度算法 | 简要说明 |
| --- | --- |
| 先来先服务（FCFS） | 按照进程到达时间顺序执行，简单但可能导致部分进程"饥饿" |
| 短作业优先（SJF） | 优先执行估计运行时间最短的作业，能显著降低平均等待时间，但可能导致长作业饥饿 |
| 高响应比优先（HRRF） | 响应比 = (等待时间 + 服务时间) / 服务时间，选择响应比最高的作业执行，兼顾等待时间与服务时间 |
| 时间片轮转（RR） | 每个进程分配一个时间片，时间用完即切换，提高系统响应性，适合分时系统 |
| 优先级调度 | 根据进程优先级高低进行调度，支持抢占/非抢占两种策略，可能出现低优先级进程饥饿问题 |
| 多级反馈队列（MLFQ） | 使用多个优先级就绪队列，进程可动态在队列间移动，结合了时间片轮转和优先级调度，适应性强 |
| 完全公平调度器（CFS） | Linux 的默认调度器，使用红黑树按"虚拟运行时间"调度进程，实现按权重"公平分配 CPU 时间" |

## 僵尸进程与孤儿进程

### 僵尸进程（Zombie Process）

定义：当一个子进程执行完毕（终止）后，其退出信息（如退出码）还没被父进程回收（通过 `wait()` 或 `waitpid()`），该子进程就进入僵尸状态，成为僵尸进程。它在进程表中保留一项记录，用于父进程获取信息。

带来的问题：僵尸进程已经终止，不会占用 CPU 和内存，但会有其他问题：

- 僵尸进程会占用系统的进程表项（PCB，用于保存退出状态），而进程表项数量是有限的。
- 如果大量僵尸进程堆积，可能会导致系统无法创建新进程，引发系统资源耗尽。
- 极端情况下可能导致系统不稳定甚至崩溃。

解决方案：

- 让父进程调用 `wait()` / `waitpid()` 回收子进程资源。
- 如果父进程忘记回收，可以杀掉父进程，让子进程变成孤儿进程，由 init（或 systemd）接管后，init 会负责回收。
- 使用信号机制（如 SIGCHLD）通知父进程子进程已退出，触发回收机制。

### 孤儿进程（Orphan Process）

定义：当一个子进程还在运行，而它的父进程已经退出，该子进程会成为孤儿进程。操作系统会自动将其交给 PID 为 1 的 init/systemd 进程进行接管和管理。

带来的问题：孤儿进程不会像僵尸进程那样占用系统资源，因为 init/systemd 会自动回收它们的状态；但如果这些孤儿进程本身是无意义的或无限循环，可能会导致占用大量 CPU 或内存资源，增加系统负担，系统性能下降。

解决方案：

- 定期检查系统中 PPID=1 的进程：`ps -ef | awk '$3==1'`（$3 表示第三列即 PPID，当该值为 1 时表示其父进程是 init）。
- 如果是异常程序导致的，应修复程序逻辑或添加守护策略。
- 可使用 kill 杀掉异常的孤儿进程。

## 进程与线程的通信

### 进程间通信方式（IPC）

| 通信方式 | 特点与说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 管道（Pipe） | 半双工，父子进程常用。匿名管道只能用于有亲缘关系的进程 | 简单易用，内核提供缓冲，适合轻量通信 | 只能单向通信；仅限父子/兄弟进程；缓冲区有限 |
| 命名管道（FIFO） | 类似匿名管道，但有名字，可用于无亲缘关系进程 | 可以在无亲缘进程间通信；简单 | 依然是半双工；数据结构简单 |
| 消息队列（Message Queue） | 内核维护消息队列，进程通过发送/接收消息通信 | 支持异步通信，结构清晰，适合复杂通信 | 内核开销大；消息体长度有限 |
| 共享内存（Shared Memory） | 将一段物理内存映射到多个进程虚拟空间中，需结合信号量或锁保证同步 | 最高效（无需内核中转）；大数据量传输性能优越 | 同步控制复杂（需加锁）；调试困难 |
| 信号量（Semaphore） | 本质是同步机制，用于控制多个进程对共享资源的访问 | 控制资源并发访问；常与共享内存搭配使用 | 只能用于同步控制；不适合数据通信 |
| 信号（Signal） | 操作系统向进程发送异步通知，如 SIGINT 等，可自定义处理函数 | 通知机制简单、轻量，适合事件通知 | 信息少，只能传递编号；不可控性强，处理麻烦 |
| 套接字（Socket） | 支持本地进程间或网络通信，基于 TCP/UDP，可用于不同主机之间 | 功能强大；支持远程通信；无亲缘进程可用 | 编程复杂；性能不如共享内存 |
| mmap 映射文件通信 | 将文件映射到多个进程地址空间中，通过读写文件实现通信 | 简单，文件可持久化 | 同步依然要靠信号量或锁；不适合高并发 |

### 线程间通信方式

线程属于同一进程，天然共享内存空间，因此通信本质上是「对共享数据的读写」+「同步机制」的组合。

| 通信方式 | 特点与说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 共享变量 + 同步机制 | 最常见方式，直接访问全局变量或堆上的共享数据，需使用互斥锁、条件变量等控制并发访问 | 通信效率极高；无内核参与 | 容易发生竞态；同步控制复杂；调试困难 |
| 条件变量（Condition） | 与互斥锁配合，支持线程在某个条件不满足时阻塞等待，满足后被唤醒 | 支持生产者-消费者模型，逻辑清晰 | 使用上较复杂，涉及原子性和死锁问题 |
| 信号量（Semaphore） | 支持多线程并发资源控制与同步，POSIX 信号量在线程间也适用 | 控制资源并发访问方便；比互斥锁灵活 | 容易误用；代码不易维护 |
| 事件（Event） | 多用于 Windows 编程，线程可等待事件触发 | 使用方便；跨线程同步 | 仅限某些平台；实现机制依赖平台 |
| 管道/队列（Queue） | 多线程通信框架中常用，如生产者线程写入、消费者线程读取（结合锁）。Python 的 queue.Queue 就是线程安全的 | 封装良好，线程安全；支持阻塞/非阻塞 | 内部实现依赖锁，仍有上下文切换 |
| 消息传递框架 | 如 Go 的 channel、Rust 的 mpsc，或 C++ 的消息总线 | 更高级抽象，避免显式加锁 | 抽象层高，理解成本略高 |

进程间通信与线程间通信对比：

| 特性 | 进程间通信 | 线程间通信 |
| --- | --- | --- |
| 通信效率 | 较低（需内核参与） | 极高（直接共享内存） |
| 通信复杂度 | 较复杂（需要多种机制支持） | 相对简单（共享变量 + 锁） |
| 通信安全性 | 高（内存隔离） | 低（需手动同步） |
| 使用场景 | 不同进程间协作、大量数据传输、网络服务等 | 同一进程内高并发处理、任务队列、异步事件处理等 |

### 锁的种类

锁是并发控制机制，用于控制多个线程/进程对共享资源的访问，防止出现竞态条件、数据不一致等问题。常见锁：

- 互斥锁（Mutex）：只能一个线程持有，用于互斥访问资源。
- 读写锁（RWLock）：支持多个读线程并发访问，但写时独占。
- 自旋锁（Spinlock）：忙等待，不释放 CPU，适合锁时间短的场景。它是一种非阻塞的互斥锁，当一个线程尝试获取锁时，如果锁已被其他线程持有，它不会挂起（阻塞）当前线程，而是会在原地不断循环（"自旋"）检查锁是否被释放。
- 递归锁：同一线程可以多次加锁，不会死锁。
- 悲观锁 / 乐观锁：悲观锁假设冲突常发生，先加锁再操作；乐观锁假设冲突少见，提交时校验（如版本号/CAS）。

## 死锁

### 死锁的概念（与饥饿、死循环的区别）

死锁（Deadlock）是指多个进程/线程因争夺资源而造成的一种互相等待的现象，即每个进程都在等待其他进程释放它自己需要的资源，从而导致所有进程都无法继续执行。

导致进程/线程无法推进的场景除了死锁还有：

- 饥饿：由于长期得不到想要的资源，某进程无法向前推进的现象。
- 死循环：某进程执行过程中一直跳不出某个循环的现象。

相同点：都是进程无法顺利向前推进的现象（故意设计的死循环除外）。

区别：

- 死锁的发生至少要有两个或两个以上的进程；饥饿的发生可能只有一个进程。
- 死锁和饥饿是管理者（操作系统）的问题，死循环是被管理者的问题（程序逻辑错误，是程序员的问题）。

### 死锁的四个必要条件

四个条件必须同时成立才会产生死锁，破坏其中任一即可避免死锁：

| 条件名称 | 含义说明 |
| --- | --- |
| 互斥 | 某资源一次只能被一个进程使用 |
| 不剥夺 | 已分配资源不能被强制剥夺，只能由进程主动释放 |
| 请求并保持 | 进程已获得部分资源，又请求其他资源但不释放已有资源 |
| 循环等待 | 存在进程组成的资源等待环，每个进程等待下一个进程所持有的资源 |

### 死锁的处理策略

| 策略 | 核心思想 | 常见方式 | 特点说明 |
| --- | --- | --- | --- |
| 预防 | 破坏死锁产生的条件之一，防患于未然 | 禁止循环等待、强制释放资源等 | 实现简单但资源利用率低 |
| 避免 | 动态检测资源分配是否会进入不安全状态，若会则不分配资源 | 银行家算法（Banker's Algorithm） | 性能消耗大，适用于静态系统 |
| 检测与解除 | 允许死锁发生，通过检测机制发现死锁并强制处理 | 周期性检测 + 撤销进程/抢占资源 | 灵活但代价大，适合容忍一定死锁的系统 |

### 预防死锁的具体方法

通过破坏死锁的四个必要条件之一或多个来从根本上避免死锁的发生：

- 破坏"互斥条件"：尽量使用可共享资源（如只读文件、缓存等），多个进程可同时访问。但实际很多资源本身无法共享（如写文件、打印机），因此此方法适用范围有限。
- 破坏"不可剥夺条件"：如果一个进程请求新资源失败，则主动释放已占资源并等待重试。系统需支持资源状态保存与恢复，适用于可中断型资源（如内存页、CPU 时间片）。
- 破坏"请求并保持条件"：进程在启动时一次性申请所需全部资源，若不能满足则等待所有资源就绪再执行。降低灵活性，但能有效防止因边申请边持有造成的资源环。
- 破坏"循环等待条件"：为所有资源统一编号，进程按照编号升序申请资源，禁止逆序申请。常用于多线程加锁场景（锁 A → 锁 B，避免锁 B → 锁 A 形成死锁链）。

### 避免死锁与解除死锁

避免死锁：

- 破坏"循环等待"：规定所有资源必须按统一顺序申请，防止环形等待。
- 破坏"占有且等待"：进程在申请新资源前必须一次性申请完，或先释放已有资源后再申请。
- 使用银行家算法：模拟每次资源分配，若发现分配后系统可能进入死锁状态，则拒绝当前请求。
- 采用锁顺序规则：多线程程序中，所有线程按照统一顺序加锁。

解除死锁：

- 终止进程：终止部分或全部死锁相关进程，释放资源，打破循环等待。可以优先终止低优先级进程或计算开销较小的进程。
- 资源剥夺：强制从部分进程中回收资源，使其他进程得以继续执行。
- 进程回滚：将某些进程恢复到以前的某个安全状态，重新执行避免死锁。

## 内存管理

### 物理地址、虚拟地址与虚拟内存

物理地址（Physical Address）：

- 定义：物理地址是指内存条上真实存在的地址，是 CPU 访问主存（RAM）时使用的地址。
- 特点：直接对应内存芯片上的某个具体位置；所有硬件最终都只能访问物理地址；操作系统负责将虚拟地址转换为物理地址（地址映射）。物理地址由内存管理单元（MMU）最终生成并用于访问主存。

虚拟地址（Virtual Address）：

- 定义：由 CPU 在执行程序时生成的地址，不直接对应物理内存，而是通过地址转换机制（如分页）被映射到实际的物理地址。
- 特点：是进程在运行时看到的地址空间；每个进程拥有独立的虚拟地址空间（从 0 开始），彼此之间不会冲突；程序中看到的所有地址默认都是虚拟地址。逻辑地址（也称为虚拟地址）是程序在运行时生成的地址，是相对于某个基地址的偏移量，由 CPU 生成，并由操作系统和 MMU 转换为物理地址。

虚拟内存：

- 定义：虚拟内存是操作系统提供的一种抽象机制，使得每个进程都像拥有一个连续且独立的大内存。
- 特点：借助硬件（MMU）和分页机制，将虚拟地址映射到物理地址；可以实现比物理内存更大的地址空间（通过磁盘作为补充）。

### 为什么需要虚拟内存

- 扩展内存容量：让程序看起来拥有比实际物理内存更大的空间（通过与磁盘交换）。
- 内存隔离：不同进程拥有各自独立的地址空间，防止相互干扰，提升安全性。
- 简化编程模型：程序可以使用从 0 开始的连续地址空间，方便开发。
- 实现按需加载：程序只在访问某段代码或数据时才将其加载到内存，节省资源。
- 支持内存共享：允许多个进程共享某些内存区域（如共享库），节省内存空间。

总结一句话：虚拟内存通过地址映射机制，为程序提供更大、更安全、更灵活的内存使用方式。

### 内部碎片与外部碎片

内存碎片是指在内存分配与回收过程中，由于内存空间不能被充分利用，导致产生了无法有效使用的空间区域，造成内存浪费。通常分为内部碎片和外部碎片两类。

内部碎片：

- 定义：分配给进程的内存区域中，未被进程实际使用的部分空间。这部分空间虽已分配，但无法用于其他进程，造成浪费。
- 特点：出现在固定大小内存分配中，如分页系统、固定分区；每个分区或页的末尾可能有"用不满"的空间。
- 举例：操作系统每次分配内存以 4KB 为单位，某个进程只用到了 3.6KB，那么剩余的 0.4KB 就是内部碎片。

外部碎片：

- 定义：内存中虽然有足够总量的空闲空间，但这些空间是分散的、非连续的，导致无法为需要大块连续空间的进程分配内存。
- 特点：出现在可变大小分配方式中，如可变分区分配；无法满足大块分配请求，尽管总空闲空间足够。
- 举例：内存中空闲块有 10KB + 15KB + 25KB，但某进程请求 40KB 连续内存，系统无法满足，形成外部碎片。

| 项目 | 内部碎片 | 外部碎片 |
| --- | --- | --- |
| 出现位置 | 分配单元内部 | 分配单元之间 |
| 产生原因 | 分配单元大于实际需求 | 空闲块过于零碎且不连续 |
| 常见于 | 分页、固定分区分配 | 可变分区、连续分配 |
| 是否可合并回收 | 否 | 是（可通过紧缩或内存整理） |

### 内存管理方式总览

| 分类 | 内存管理方式 | 简要说明 |
| --- | --- | --- |
| 早期方式 | 连续内存分配 | 整个程序被加载到连续的物理内存空间中，简单但易产生碎片 |
| 早期方式 | 固定分区分配 | 将内存划分为固定大小的多个区域，适用于早期批处理系统，可能浪费较大内存 |
| 早期方式 | 可变分区分配 | 分区大小可变，使用首次适应、最佳适应等算法进行分配，但容易产生外部碎片 |
| 中期方式 | 分页（Paging） | 将内存与程序划分为固定大小的页和页框，按页映射，消除外部碎片，但有内部碎片 |
| 中期方式 | 分段（Segmentation） | 根据程序逻辑结构划分内存段（如代码段、数据段等），更贴近编程语义，但可能有碎片 |
| 现代方式 | 段页式管理 | 先分段再分页，结合分页与分段优点，是现代操作系统（如 Linux）主流方式 |
| 现代方式 | 虚拟内存管理 | 支持按需加载和页面置换机制，不需一次性加载程序全部内容，节省物理内存 |

注意：虚拟内存管理和段页式管理在一个现代操作系统中是可以同时存在的，它们是互补关系，并不是互斥的概念。

### 分页 vs 分段 vs 段页式

注意：在一个进程中，段表只有一个，而页表可能有多个。

| 对比项 | 分页 | 分段 | 段页式 |
| --- | --- | --- | --- |
| 本质单位 | 信息的物理单位 | 信息的逻辑单位 | 混合单位（逻辑 + 物理） |
| 划分方式 | 固定大小页划分 | 按逻辑模块划分 | 先分段再分页 |
| 目的 | 为系统管理所需，提高内存利用率 | 为更好满足用户需求 | 兼顾系统管理与用户需求 |
| 单位大小 | 固定（如 4KB） | 不固定 | 段大小不固定，页大小固定 |
| 长度决定者 | 系统固定 | 用户程序决定 | 段由用户决定，页由系统固定 |
| 地址结构 | 页号 + 页内偏移 | 段号 + 段内偏移 | 段号 + 页号 + 页内偏移 |
| 作业地址空间维度 | 一维 | 二维 | 三维（段 + 页 + 页内偏移） |
| 是否支持非连续物理内存 | 是 | 否 | 是 |
| 碎片问题 | 有内部碎片，无外部碎片 | 无内部碎片，有外部碎片 | 内外碎片都较少，综合效果好 |
| 管理机制 | 页表 | 段表 | 段表 + 多级页表 |
| 存在数量 | 一个进程可有多个页表 | 一个进程只有一个段表 | 每段有对应页表 |

## 栈与堆

### 栈空间与堆空间

栈空间（Stack）是由编译器自动分配和释放的一块内存区域，用于存储局部变量、函数参数、返回地址等信息：

- 遵循先进后出（LIFO）原则。
- 每次函数调用时会在栈上分配空间，函数返回时自动回收。
- 由操作系统管理，大小有限（如 1MB 或几 MB）。
- 访问速度快，适合存储生命周期短的变量。

堆空间（Heap）是由程序员手动分配和释放的内存区域（如 malloc、new），用于动态分配内存，生命周期由程序控制：

- 空间大、灵活。
- 不会自动释放，需手动释放（如 free、delete）。
- 容易产生内存泄漏或碎片。
- 通常适合存储生命周期较长或大小不确定的对象。

| 对比项 | 栈空间（Stack） | 堆空间（Heap） |
| --- | --- | --- |
| 管理方式 | 操作系统自动分配/回收 | 程序员手动申请和释放 |
| 分配效率 | 高（维护简单，顺序分配） | 低（维护复杂，需查找合适块） |
| 生命周期 | 随函数调用自动创建和释放 | 由程序控制，可能较长 |
| 空间大小 | 较小（通常几 MB） | 较大（受限于物理内存） |
| 是否产生碎片 | 不产生（连续分配） | 容易产生（频繁分配释放） |
| 常见问题 | 栈溢出（调用层级深、递归过多） | 内存泄漏、碎片、悬垂指针 |
| 典型用途 | 局部变量、函数调用上下文 | 动态结构（链表、树、对象等） |
| 是否支持逃逸分析优化 | 是，可避免不必要的堆分配 | 无，必须由程序控制释放 |

### 逃逸分析（Escape Analysis）

逃逸分析是一种由编译器在编译期进行的优化技术，用来分析变量的作用域，从而判断其是否可以在栈上分配而不逃逸到堆：

- 如果变量不会逃出当前函数（即在外部不可访问），则可将其分配在栈上。
- 如果变量被返回或被外部引用，则它会逃逸到堆中。
- 常见于 Go、Java 等支持栈上分配优化的语言。
- 可减少 GC 压力，提升性能。

常见的逃逸场景：

- 返回局部变量的指针：返回局部变量地址时，该变量必须逃逸到堆上，因为其生命周期超出了函数作用域。
- 闭包引用外部变量：闭包可能在函数返回后继续使用局部变量，因此这些变量必须逃逸到堆上，以保证数据在闭包使用时仍然有效。
- 将变量赋值给接口：变量被赋值给接口（如 interface{}）时，由于接口是一个动态类型结构，编译器通常会将变量分配到堆上以保证其可访问性。
- 将变量作为引用参数传递给函数：如果函数可能将传入的引用保存到堆结构中（如全局变量、闭包等），那么被引用的变量需要逃逸到堆。
- 切片或 map 导致内部对象逃逸：虽然切片或 map 本身可在栈上，但若对其元素取地址、或发生扩容，底层数据可能逃逸到堆中。

## 内存问题

### 内存泄漏、内存溢出与内存逃逸

内存泄漏：

- 定义：程序中已经不再使用的内存未被释放，导致这部分内存无法再被使用，但仍然占用资源。
- 表现：随着程序运行，内存占用不断增加，最终可能导致系统变慢甚至崩溃。
- 常见原因：长生命周期对象持有短生命周期对象的引用；忘记 free()（C/C++）；容器未清空（如 Java 的集合类）；回调函数未注销；资源未关闭（如未关闭的流或连接）。

内存溢出：

- 定义：程序申请内存失败，系统无法提供足够的内存资源，导致程序运行异常甚至崩溃。
- 表现：Java 抛出 OutOfMemoryError；C/C++ 返回 NULL；系统宕机或被 OOM Killer 杀掉。
- 常见原因：无限创建对象或分配内存；缓存或数据未及时清理；死循环中持续分配内存。

内存逃逸：

- 定义：指本应分配在栈上的变量被"逃逸"到了堆上，导致其生命周期超出原本的作用域。
- 关键点：与编译器的逃逸分析优化有关；会增加 GC 压力，影响性能。
- 常见于：返回局部对象的指针或引用；将局部变量赋值给全局变量或堆对象；被闭包引用（如 Go 中的闭包变量）。

### 三者对比与 GC 辨析

| 项目 | 内存泄漏 | 内存溢出 | 内存逃逸 |
| --- | --- | --- | --- |
| 定义 | 占用内存不释放 | 内存不足以满足申请 | 本应在栈上却被分配到堆上 |
| 危害 | 内存浪费，可能导致溢出 | 程序崩溃 | 增加 GC 压力，降低性能 |
| 是否错误 | 是 | 是 | 不是错误，但影响性能 |
| 关注点 | 资源释放 | 内存使用总量 | 逃逸分析优化 |

有垃圾回收机制，为什么还会出现内存泄漏、溢出、逃逸？

| 问题 | 简要原因说明 |
| --- | --- |
| 内存泄漏 | GC 只能回收无引用对象，如果仍保留无用对象的引用，GC 无法释放，仍然泄漏 |
| 内存溢出 | 程序申请内存超过系统或虚拟机限制，即使 GC 正常工作，也可能因分配过多导致 OOM |
| 内存逃逸 | 是编译器层面的问题，与 GC 无关，本应分配在栈上的变量因逃逸行为被分配到了堆上，增加 GC 负担 |

总结一句话：GC 只是自动帮你清理"无用"内存，但无法阻止你写出"浪费内存"的代码。滥用内存、引用不清、逃逸设计不当仍会出问题。

## I/O 模型

### I/O 基础与网络 I/O 阻塞

I/O（Input/Output）是指计算机系统中输入和输出设备与 CPU/内存之间的数据交换操作，包括：

- 磁盘 I/O（读写文件）
- 网络 I/O（数据包的发送和接收）
- 终端 I/O（用户输入输出）

在操作系统中，I/O 通常是慢速操作，CPU 需等待设备完成，因此 I/O 效率直接影响系统性能。

网络 I/O 阻塞：网络 I/O 本质上涉及"等待数据准备好"的过程，比如等待对方发送的数据到达（recv）、等待系统内核准备好发送缓冲区（send）。阻塞的原因主要是：

- 调用系统调用时数据还没准备好（如 read() 没有可读数据，或 write() 缓冲区满）。
- 内核出于安全性和资源管理考虑，不让进程忙等，而是挂起等待 I/O 事件。

在默认情况下，网络套接字是阻塞模式，系统调用会挂起进程直到数据就绪。

### 同步/异步与阻塞/非阻塞

同步 vs 异步——关注的是消息通信机制，关键在于调用者是否等待结果返回：

| 模型 | 定义 |
| --- | --- |
| 同步（Synchronous） | 调用者必须等待被调用者处理完并返回结果后，调用才算完成 |
| 异步（Asynchronous） | 调用者不等待被调用者处理完，而是立刻返回，结果稍后再通过回调、事件或消息通知 |

举例：同步是你去餐厅点餐，等菜做好端上来你才开动；异步是你点了餐就回去干活了，菜做好了服务员会通知你来吃。

阻塞 vs 非阻塞——关注的是程序在等待调用结果时的状态，即当前线程在等待时是否会被挂起：

| 模型 | 定义 |
| --- | --- |
| 阻塞（Blocking） | 当前线程调用某个操作时，会挂起（休眠）等待结果返回，这段时间不能干别的 |
| 非阻塞（Non-blocking） | 当前线程调用时不会等待结果，立即返回，并可以继续做其他事，需要轮询或回调拿结果 |

组合对比——同步/异步看调用者是否等待结果，阻塞/非阻塞看线程是否挂起休眠。把"同步=阻塞"、"异步=非阻塞"直接画等号是错误的简化：

| 实际组合 | 是否存在 | 举例说明 |
| --- | --- | --- |
| 同步 + 阻塞 | 最常见 | 普通的函数调用、网络 recv() |
| 同步 + 非阻塞 | 存在 | select()、poll() 等返回立即值，但你还是要自己同步去取 |
| 异步 + 阻塞 | 特殊场景 | 异步调用后手动等待回调完成 |
| 异步 + 非阻塞 | 经典模型 | 回调、事件驱动模型，如 Node.js 网络编程 |

用去银行办理业务类比：

| I/O 模型 | 类比场景 |
| --- | --- |
| 阻塞 I/O | 去排队窗口，站着等办好业务才能离开 |
| 非阻塞 I/O | 去窗口问是否轮到你，没轮到就先走开，过会再来问一次（轮询） |
| 同步 I/O | 无论站着等还是来回跑，你自己总得等待结果（轮询/挂起） |
| 异步 I/O | 办理业务交给柜员，你先去喝咖啡，业务好了他们通知你 |

### 五种常见的 I/O 模型

常见的五种 I/O 模型分别是：阻塞 I/O、非阻塞 I/O、I/O 复用（select、poll、epoll）、信号驱动 I/O 和异步 I/O。它们的核心区别在于应用程序等待数据准备与数据拷贝的方式不同。其中最常用于高并发场景的是 I/O 复用（特别是 epoll），而真正意义上的异步 I/O 则效率最高，但实现与支持较复杂。

| I/O 模型 | 特点说明 | 常见应用场景 |
| --- | --- | --- |
| 阻塞 I/O（Blocking I/O） | 默认方式，read() 等系统调用会阻塞直到数据准备好并复制完成 | 最简单，效率最低（传统 socket） |
| 非阻塞 I/O（Non-blocking I/O） | read() 不会阻塞，数据没准备好会立刻返回错误（如 EAGAIN），需反复轮询 | 一般用于轮询式多路通信 |
| I/O 复用（Select/Poll/Epoll） | 通过 select/poll/epoll 等统一等待多个 fd 的 I/O 事件，避免阻塞在某一个上 | 高并发服务端（如 Nginx） |
| 信号驱动 I/O（Signal-driven I/O） | 注册信号处理函数（如 SIGIO），内核 I/O 就绪后发信号通知用户程序 | 较少使用，复杂性高 |
| 异步 I/O（Asynchronous I/O, AIO） | 提交 I/O 请求后立即返回，操作系统完成后通知应用程序（回调、事件） | 高性能场景，RT 系统、Windows、io_uring |

一句话区分：阻塞/非阻塞/复用/信号驱动的 I/O 是"自己动手等"；异步 I/O 是"交给系统办，办完告诉我"。

### I/O 模型特性对比与发展趋势

| 模型 | 阻塞 I/O | 非阻塞 I/O | I/O 复用 | 信号驱动 I/O | 异步 I/O |
| --- | --- | --- | --- | --- | --- |
| 等待阶段是否阻塞 | 是 | 否 | 是（统一等待） | 否（信号通知） | 否（交由系统完成） |
| 是否需轮询 | 否 | 是 | 部分是（select/poll） | 否 | 否 |
| 系统开销 | 高（多线程） | 高（频繁系统调用） | 中（epoll 最好） | 中（复杂） | 低（高效，需支持） |
| 编程复杂度 | 低 | 中 | 中（epoll 高效） | 高 | 高 |
| 常见场景 | 教学、早期模型 | 多路复用底层 | Web 服务器、高并发 | 极少见 | 高性能系统，如数据库 |

模型发展趋势：阻塞 I/O → 非阻塞 I/O → I/O 复用（select → poll → epoll）→ 信号驱动 I/O（支线分支，使用较少）→ 异步 I/O（AIO、io_uring、IOCP 等）。

| 阶段 | 问题 / 局限性 | 目标改进方向 |
| --- | --- | --- |
| 阻塞 I/O | 每个请求一个线程，线程被长时间挂起 | 减少线程数，提高资源利用率 |
| 非阻塞 I/O | 频繁轮询系统调用，CPU 浪费 | 尝试集中管理多个 I/O 事件 |
| I/O 复用 | select/poll 扫描性能差 | 引入 epoll，提升事件通知效率 |
| 信号驱动 I/O | 逻辑复杂、可靠性差 | 被 I/O 复用与异步 I/O 替代 |
| 异步 I/O | 完全交由内核处理，应用专注业务逻辑 | 真正的高并发、低延迟 I/O 处理 |

当前主流 I/O 模型：

| 系统平台 | 推荐模型 | 技术 |
| --- | --- | --- |
| Linux | I/O 复用（epoll）→ io_uring | Nginx、MySQL、Redis |
| Windows | 异步 I/O（IOCP） | .NET、SQL Server |
| 高并发框架 | 统一采用事件驱动模型 | Netty、libuv、Go runtime |

说明：Nginx、MySQL、Redis 这些主流服务在 Linux 平台上采用了 I/O 复用模型（如 epoll）来实现高性能的并发处理。虽然它们都可以在 Windows 上运行，但其高性能 I/O 机制（如 epoll、io_uring）主要针对 Linux 设计，生产环境仍推荐在 Linux 上部署以获得最佳性能。

### select、poll 和 epoll 的区别

select、poll、epoll 都是 I/O 多路复用机制，用于同时监听多个文件描述符（FD），但在实现和性能上有所不同：

| 机制 | 平台 | FD 存储方式 | 是否有事件通知 | 查找复杂度 | 性能 | 适用场景 |
| --- | --- | --- | --- | --- | --- | --- |
| select | 早期 Unix、Linux、Windows | 位图（bitmap） | ❌ 需遍历所有 FD | O(n) | 低，FD 多时性能差 | 少量 FD（<1024） |
| poll | Unix、Linux | 链表 | ❌ 仍需遍历所有 FD | O(n) | 低，FD 多时仍性能差 | 中等数量 FD |
| epoll | Linux（2.6+） | 红黑树 + 事件驱动 | ✅ 事件触发（回调机制） | O(1) | 高，适合高并发 | 大规模并发（如 Redis、Nginx） |

## 用户态与内核态

### 用户态与内核态的概念

用户态（User Mode）：

- 指用户进程执行普通应用程序代码时所处的状态。
- 程序运行权限受限，不能直接操作硬件或访问内核空间。
- 一旦需要进行 I/O、文件操作、进程控制等特权操作，必须发起系统调用，切换到内核态。

内核态（Kernel Mode）：

- 指 CPU 执行操作系统核心代码（kernel）时所处的状态。
- 拥有最高权限，可以直接操作硬件、管理内存、调度进程。
- 执行对象包括：系统调用处理、设备驱动、中断处理、进程调度等。

### 为什么区分用户态和内核态、切换时机与代价

区分的原因：

| 原因 | 说明 |
| --- | --- |
| 系统安全性 | 防止普通程序恶意或无意间直接访问硬件或修改内核数据，保障系统稳定 |
| 稳定性隔离 | 应用程序出错（如野指针）只能影响自身，而不会导致整个操作系统崩溃 |
| 资源保护 | 操作系统需要控制资源（CPU、内存、I/O），防止进程随意抢占或修改 |
| 特权操作管控 | 某些操作如创建进程、写磁盘、网络收发，必须通过内核，统一调度和管理 |

简单理解：用户态 = 普通人；内核态 = 超级管理员。安全起见，不能让普通人拿管理员权限乱改系统。

用户态切换到内核态的典型时机：

| 场景 | 描述 |
| --- | --- |
| 系统调用 | 如 read()、write()、fork()，主动请求内核服务 |
| 软中断 | 比如进程调度、中断处理程序 |
| 异常 | 如非法内存访问、除零错误，触发 CPU 异常 |
| 页错误（Page Fault） | 访问不存在的虚拟页，需要内核处理缺页 |

切换代价：切换涉及模式切换 + 上下文保存/恢复 + TLB flush，通常代价较大，频繁切换会降低性能（所以系统调用要尽量精简）。

## 并发并行与程序执行过程

### 并发与并行

并发（Concurrency）和并行（Parallelism）都涉及多个任务的执行，但关注点不同：

| 对比项 | 并发（Concurrency） | 并行（Parallelism） |
| --- | --- | --- |
| 定义 | 同一时间段内多个任务交替进行 | 同一时刻，多个任务同时真正进行 |
| 本质 | 逻辑上的同时 | 物理上的同时 |
| 场景 | 单核 CPU 上多任务切换 | 多核 CPU 上任务同时运行 |
| 举例 | 多线程交替打印数字 | 多线程分别处理不同任务同时进行 |

类比：并发像一个服务员同时接多个桌的订单，来回切换服务；并行则像多个服务员分别服务各桌，真正同时进行。

在操作系统中：并发依赖于进程/线程的切换机制；并行依赖于多核 CPU 的硬件能力。现代系统通常同时使用并发和并行来提高性能。

### 程序的完整执行过程

一个程序从运行到结束，大致可以分为以下几个阶段：

1. 加载（Loading）：用户通过命令或点击图标启动程序，操作系统通过程序的可执行文件（如 ELF 或 PE）创建进程，并将其代码段、数据段加载到内存。
2. 创建进程（Process Creation）：操作系统为其分配唯一的 PID，初始化用户栈、堆、寄存器，建立虚拟地址空间，创建 PCB（进程控制块）。
3. 调度执行（Scheduling & Execution）：进程被加入就绪队列，等待 CPU 调度。CPU 调用调度器选择此进程后开始执行，其指令被逐条执行。若有系统调用（如读取文件、网络通信），会陷入内核态执行对应功能。
4. 状态切换（State Transition）：进程可能因 I/O 阻塞、时间片用完、主动让出 CPU 等进入等待或就绪状态。操作系统维护调度队列，决定下次运行哪个进程。
5. 终止（Termination）：程序执行完毕或被终止（exit/kill），操作系统回收资源（内存、文件句柄等），更新 PCB 状态为退出，最终清除该进程。
