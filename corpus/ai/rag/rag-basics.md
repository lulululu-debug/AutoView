# 大模型 RAG（基础概要）

## RAG 出现的原因

- 知识的局限性：模型自身的知识完全源于它的训练数据，而现有的主流大模型（ChatGPT、文心一言、通义千问等）的训练集基本都构建于网络公开的数据，对于一些实时性的、非公开的或离线的数据是无法获取到的，这部分知识也就无从具备。
- 幻觉问题：所有 AI 模型的底层原理都是基于数学概率，其模型输出实质上是一系列数值运算，大模型也不例外，所以它有时候会一本正经地胡说八道，尤其是在大模型自身不具备某一方面的知识或不擅长的场景。这种幻觉问题的区分是比较困难的，因为它要求使用者自身具备相应领域的知识。
- 数据安全性：对于企业来说，数据安全至关重要，没有企业愿意承担数据泄露的风险，将自身的私域数据上传第三方平台进行训练。这也导致完全依赖通用大模型自身能力的应用方案不得不在数据安全和效果方面进行取舍。

## RAG 发展阶段

Yunfan Gao 等人在《Retrieval-Augmented Generation for Large Language Models: A Survey》中详细考察了 RAG 范式的演变和发展，将其分成三个阶段：朴素 RAG、高级 RAG 和模块化 RAG。三个范式之间是继承与发展的关系：Advanced RAG 是 Modular RAG 的一种特例形式，而 Naive RAG 则是 Advanced RAG 的一种特例。

### 朴素 RAG（Naive RAG）

朴素 RAG 最早出现，包括索引、检索和生成三部分：

- 索引：将文档库分割成较短的 Chunk，并通过编码器构建向量索引。
- 检索：根据问题和 chunks 的相似度检索相关文档片段。
- 生成：以检索到的上下文为条件，生成问题的回答。

朴素 RAG 简单易懂，但存在多个挑战：

- 检索阶段：精确性和召回率往往是一个难题，既要避免选择无关片段，又要避免错过关键信息。
- 信息整合：面对复杂问题，单个检索可能不足以获取足够的上下文信息；对检索的结果，要确定段落的重要性和相关性，对段落进行排序，并对冗余段落进行处理。
- 生成阶段：模型可能会面临幻觉问题，即产生与检索到的上下文不符的内容；模型可能会过度依赖上下文信息，导致只生成检索到的内容而缺乏自己的见解；同时要尽量避免模型输出不相关、有毒或有偏见的信息。

### 高级 RAG（Advanced RAG）

为了解决朴素 RAG 遗留的问题，高级 RAG 引入了改进措施，增加了预检索过程（Pre-Retrieval Process）和后检索过程（Post-Retrieval Process）两个阶段来提高检索质量。

预检索过程主要关注索引优化（index optimization）和查询优化（query optimization）：

- 索引优化的目标是提高被索引内容的质量，常见方法有：提高数据粒度（enhancing data granularity）、优化索引结构（optimizing index structures）、添加元数据（adding metadata）、对齐优化（alignment optimization）和混合检索（mixed retrieval）。
- 查询优化的目标是使用户的原始问题更清晰、更适合检索任务，常见方法有：查询重写（query rewriting）、查询转换（query transformation）、查询扩展（query expansion）等技术。

后检索过程关注如何将检索到的上下文有效地与查询整合起来。直接将所有相关文档输入大模型可能会导致信息过载，使关键细节与无关内容混淆。为了减轻这种情况，后检索过程引入的方法包括：

- 重新排序块（rerank chunks）
- 上下文压缩（context compressing）
- 上下文筛选缩短窗口长度等

尽管高级 RAG 在检索前和检索后提出了多种优化策略，但它仍然遵循着和朴素 RAG 一样的链式结构，架构的灵活性仍然受到限制。

### 模块化 RAG（Modular RAG）

模块化 RAG 是一种将 RAG 系统的不同功能组件模块化的设计方式，使系统更易于管理、扩展和优化。模块化 RAG 的架构超越了前两种 RAG 范式，增强了其适应性和功能性，可以灵活地引入特定功能模块或替换现有模块，整个过程不仅限于顺序检索和生成，还包括迭代和自适应检索等方法。

## RAG 应用流程

### 数据准备阶段

完整的 RAG 应用流程主要包含两个阶段：数据准备阶段（数据提取 → 文本分割 → 向量化 embedding → 数据入库）和应用阶段（用户提问 → 数据检索/召回 → 注入 Prompt → LLM 生成答案）。

数据准备一般是一个离线的过程，主要是将私域数据向量化后构建索引并存入数据库的过程：

数据提取：

- 数据加载：包括多格式数据加载、不同数据源获取等，根据数据自身情况，将数据处理为同一个范式。
- 数据处理：包括数据过滤、压缩、格式化等。
- 元数据获取：提取数据中关键信息，例如文件名、Title、时间等。

文本分割主要考虑两个因素：1）embedding 模型的 Tokens 限制情况；2）语义完整性对整体检索效果的影响。常见的文本分割方式：

- 句分割：以"句"的粒度进行切分，保留一个句子的完整语义。常见切分符包括句号、感叹号、问号、换行符等。
- 固定长度分割：根据 embedding 模型的 token 长度限制，将文本分割为固定长度（例如 256/512 个 tokens），这种切分方式会损失很多语义信息，一般通过在头尾增加一定冗余量来缓解。

向量化（embedding）是将文本数据转化为向量矩阵的过程，会直接影响后续检索的效果。常见的 embedding 模型：

| 名称 | 描述 |
| --- | --- |
| ChatGPT-Embedding | 主要是 text-embedding-3-small 和 text-embedding-3-large，由 OpenAI 提供，以接口形式调用 |
| Qwen3 Embedding | 阿里开发的多语言 Embedding 模型系列，支持 100+ 语言，在中文和代码检索表现优异，完全开源（Apache 2.0），支持本地部署 |
| ERNIE-Embedding V1 | 由百度提供，依赖于文心大模型能力，以接口形式调用 |
| M3E | 功能强大的开源 Embedding 模型，包含 m3e-small、m3e-base、m3e-large 等多个版本，支持微调和本地部署 |
| BGE | 由北京智源人工智能研究院发布的开源 Embedding 模型，包含支持中文和英文的多个版本，支持微调和本地部署 |
| Gemini Embedding / EmbeddingGemma | Google 官方 Embedding 模型系列 |

这些模型基本能满足大部分需求，但对于特殊场景（例如涉及罕见专有词）或想进一步优化效果，可以选择开源 Embedding 模型微调或直接训练适合自己场景的 Embedding 模型。

数据入库：数据向量化后构建索引并写入数据库。适用于 RAG 场景的数据库包括 FAISS、Chromadb、ES、Milvus 等，一般根据业务场景、硬件、性能需求等多因素综合考虑，选择合适的数据库。

### 应用阶段

流程：用户提问 → 数据检索（召回）→ 注入 Prompt → LLM 生成答案。根据用户的提问，通过高效的检索方法召回与提问最相关的知识并融入 Prompt，大模型参考当前提问和相关知识生成相应的答案。

- 用户提问：用户通过自然语言的方式向系统提出问题或查询，可以是开放性的，也可以是具体的。
- 数据检索：常见的数据检索方法包括相似性检索（语义检索、密集检索）、全文检索（关键词检索、稀疏检索）等，根据检索效果，一般可以选择多种检索方式融合，提升召回率。密集检索即计算查询向量与所有存储向量的相似性得分，返回得分高的记录，常见的相似性计算方法包括余弦相似性、欧氏距离、曼哈顿距离等；稀疏检索（全文检索）在数据存入时通过关键词构建倒排索引，在检索时通过关键词进行全文检索找到对应记录。
- 注入 Prompt：Prompt 作为大模型的直接输入，是影响模型输出准确率的关键因素之一。检索到相关数据后，系统会将这些数据作为 Prompt 注入到 LLM 中，为 LLM 提供额外的上下文信息，使得 LLM 在生成答案时能够参考这些信息，从而生成更加准确、相关的答案。
- LLM 生成答案：LLM 基于其语言生成能力，结合 Prompt 中的信息，生成并输出最终的答案。

## RAG 优化方法

### 数据处理优化

不管 RAG 系统结构怎样复杂，由于其数据驱动的特性，高信噪比的数据仍然是十分重要的。在检索之前对原始数据的优化方法：

- 实体解析：消除实体和术语的歧义以实现一致的引用。例如，将"LLM"、"大语言模型"和"大模型"标准化为通用术语。
- 增加元数据：增加内容摘要、时间戳、用户可能提出的问题等附加信息来丰富知识库。
- 数据增强：使用同义词、释义甚至其它语言的翻译来增加知识库的多样性。
- 文档划分：合理地划分不同主题的文档。如果人类都不能轻松地判断出需要查阅哪个文档才能回答提问，那么检索系统也无法做到。
- 处理特殊数据：例如时间敏感数据，对于经常更新的主题，实施一种机制来使过时的文档失效或更新。

### 文本分割优化

被检索知识库中的数据量通常远超 LLM 所能接受的输入长度，因此合理的分块（Chunking）应尽可能做到在不超出 LLM 输入长度限制的情况下，保证块之间的差异性和块内部的一致性。高级的分块方法：

- 句分割：使用 NLTK 或者 spaCy 库提供的句子分割功能，主流开发框架如 langchain 都有集成。
- 递归分割：通过重复地应用分块规则来递归地分解文本。例如在 langchain 中会先通过段落换行符（`\n\n`）进行分割，然后检查这些块的大小，如果不超过阈值则保留；对于超过阈值的块，使用单换行符（`\n`）再次分割，以此类推，不断根据块大小更新更小的分块规则（如空格、句号）。这种方法可以灵活地调整块的大小，对于密集信息部分用更细的分割捕捉细节，信息较少的部分用更大的块。
- 语义分割：通过计算向量化后的文本的相似度来进行语义层面的分割。
- 特殊结构分割：针对特定结构化内容（例如 Markdown、LaTeX、JSON 等）的专门分割器，特别设计来处理这些类型的文档，确保正确地保留其结构。

块的大小是一个重要因素：除了嵌入模型，文档的类型和用户查询的长度及复杂性也是决定分块大小的重要因素。处理长篇文章或书籍时，较大的分块有助于保留更多的上下文和主题连贯性；对于社交媒体帖子，较小的分块可能更适合捕捉每个帖子的精确语义。如果用户的查询通常简短和具体，较小的分块可能更合适；查询较复杂则可能需要更大的分块。实际场景中需要不断实验调整，在一些测试中 128 大小的分块往往是最佳选择，无从下手时可以从这个大小作为起点进行测试。

### 向量化（embedding）优化

嵌入阶段可以优化的点：

- 尽量使用动态嵌入：动态嵌入相较于静态嵌入更能够处理一词多义的情况。例如"我买了一张光盘"里"光盘"指具体的圆形盘片，而"光盘行动"中"光盘"指把餐盘里的食物吃光。语义完全不一样的词使用静态嵌入其向量是固定的；相比之下，引入自注意力机制的模型（如 BERT）能够提供动态的词义理解，根据上下文动态地调整词义，使得同一个词在不同语境下有不同的向量表示。
- 微调嵌入：大多数嵌入模型都是在通用语料上训练的，有些项目为了让模型对垂直领域的词汇有更好的理解，会对嵌入模型进行微调，使模型能够对垂直领域词汇和通用词汇一视同仁。
- 混合嵌入：对用户问题和知识库文本使用不同的嵌入模型。

### 查询优化（用户提问环节）

实际环境中，用户的表述可能多样或模糊，导致检索阶段召回率和准确率较低，这时需要对查询做优化，规范和丰富查询所包含的信息：

- 查询重写：通过提示 LLM 或者使用专门的"问题重写器"（通常是经过微调的小型 Transformer）来对用户的问题进行改写。
- 后退提示（Step-back Prompting）：提示 LLM 提出一个关于高层次概念或原则的抽象通用问题（"后退"问题），后退问题和原始问题一起进行检索。例如对于问题"Estella Leopold 在 1954 年 8 月至 11 月期间上了哪所学校？"，由于有时间范围的详细限制很难直接解决，提出后退问题"Estella Leopold 的教育经历是怎样的？"则有助于系统的检索。
- Follow Up Questions / 查询问题压缩：使用 LLM 针对历史对话和当前问题生成一个独立问题。主要针对：a. 后续问题建立在前一次对话的基础上或引用了前一次谈话，例如用户先问"我在意大利能做什么"，然后问"那里有什么类型的食物"——如果只嵌入后一句，LLM 就不知道"那里"在哪里；b. 嵌入整个对话（或最后 k 条消息）时，如果后续问题与之前的对话完全无关，可能返回完全无关的结果，从而在生成过程中分散 LLM 的注意力。
- HyDE：用 LLM 生成一个"假设"答案，将其和问题一起进行检索。核心思想是接收用户提问后，先让 LLM 在没有外部知识的情况下生成一个假设性的回复，然后将这个假设性回复和原始查询一起用于向量检索。假设回复可能包含虚假信息，但蕴含着 LLM 认为相关的信息和文档模式，有助于在知识库中寻找类似的文档。
- 多问题查询：基于原始问题，提示 LLM 从不同角度产生多个新问题或者子问题，并使用每一个新问题进行检索，在后续阶段使用 RRF 或者 rerank 合并来自不同问题的检索结果。例如对于原始问题"谁最近赢得了总冠军，红袜队还是爱国者队？"，可以生成两个子问题分别检索两队上次夺冠时间。

### 检索优化

检索（Retrieval）的最终目标是获取最相关的文档，或者保证最相关的文档在获取的文档列表中存在。优化方法：

- 上下文压缩：当文档块过大时可能包含太多不相关的信息，传递这样的文档块可能导致更昂贵的 LLM 调用和更差的响应。上下文压缩通过 LLM 的帮助根据上下文对单个文档内容进行压缩，或者对返回结果进行过滤仅返回相关信息。
- 句子窗口搜索：文档块太小会导致上下文缺失，解决方案之一是窗口搜索——当提问匹配好文档块后，将该文档块周围的块作为上下文一并交给 LLM，增加 LLM 对文档上下文的理解。
- 父文档搜索：先将文档分为尺寸更大的主文档，再把主文档分割为更短的子文档两个层级，用户问题与子文档匹配，然后将该子文档所属的主文档发送给 LLM。
- 自动合并：在父文档搜索上更进一步的复杂方案。先对文档进行结构切割，比如按三层树状结构切割：顶层节点块大小 1024、中间层 512、底层叶子节点 128。检索时只拿叶子节点和问题进行匹配，当某个父节点下的多数叶子节点都与问题匹配，则将该父节点作为结果返回。
- 混合检索：为了获得开放式用户查询的高事实召回率，通过混合多个检索方法实现不同检索技术的协同作用。例如采用向量检索 + 关键词检索的组合来构建 RAG 系统的检索模块。
- 路由机制：当建立了多个针对不同数据类型和查询需求的索引后（例如一个索引专门处理摘要类问题，另一个应对具体答案的问题，还有一个针对需要考虑时间因素的问题），需要使用路由机制来选择最合适的索引进行数据检索，提升检索质量和响应速度。
- 使用 Agent：使用 Agent 来决定应该采用什么样的检索方法，从不同的检索方法中选取一种或多种进行召回，组合方式可以是垂直关系或平行关系。例如对于查询"最新上映的科幻电影推荐"，Agent 可能首先将其路由至专门处理当前热点话题的索引，然后利用专注于娱乐和影视内容的索引来生成相关推荐。

### 检索后处理（重排序）

检索后处理是对检索结果进行进一步的处理以便于后续 LLM 更好的生成，比较典型的就是重排序（Rerank）。向量检索其实是计算语义层面的相似性，但语义最相似并不总是代表最相关。重排模型通过对初始检索结果进行更深入的相关性评估和排序，确保最终展示给用户的结果更加符合其查询意图。

实现重排序除了可以提示 LLM 进行重排，更多的是使用专门的重排序模型（例如闭源的 Cohere，开源的 BAAI 和 IBM 发布的模型）。这些模型会考虑更多的特征，如查询意图、词汇的多重语义、用户的历史行为和上下文信息，从而保证最相关的文档排在结果列表的最前面。

### 生成优化

生成（Generation）阶段的优化更多的是考虑用户体验：

- 多轮对话：也就是带聊天历史的 RAG。以 AI 搜索为例，明星产品 Perplexity 就支持多轮对话，用户可以通过连续对话来深入了解某个问题。
- 增加追问机制：在 prompt 中加入"如果无法从背景知识回答用户的问题，则根据背景知识内容，对用户进行追问，问题限制在 3 个以内"。这个机制主要依靠大模型的能力，但大大改善了用户体验，用户在多轮引导中逐步明确自己的问题，从而得到合适的答案。
- Prompt 优化：RAG 系统中的 prompt 应明确指出回答仅基于搜索结果，不要添加任何其他信息。例如："你是一名智能客服。你的目标是提供准确的信息，并尽可能帮助提问者解决问题。你应保持友善，但不要过于啰嗦。请根据提供的上下文信息，在不考虑已有知识的情况下，回答相关查询。"也可以根据场景需要适当让模型的回答融入一些主观性或其对知识的理解。此外，使用 Few-shot 的方法指导 LLM 如何利用检索到的知识，也是提升 LLM 生成内容质量的有效方法。
- 用户反馈循环：基于现实世界用户的反馈不断更新数据库，标记它们的真实性。

## RAG 的几种变体

### CRAG 与 Self-RAG

CRAG（Corrective Retrieval Augmented Generation，纠正性检索增强生成）：主要关注在检索阶段后引入一个额外的知识精炼（Knowledge Correction）环节。这个环节会对检索到的每一个相关切片（snippets）进行评估，判断其是否对问题有效。如果评估结果为不正确（Incorrect），则会丢弃原来的文档，并可能通过网络搜索获取新的相关信息。CRAG 的特色在于其自我评估和自我修正的能力，旨在提高生成结果的准确性和质量。

Self-RAG（Self-Reflective Retrieval Augmented Generation，自我反思式检索增强生成）：强调通过自我反思能力来调整自身的检索和生成结果。它会在生成内容的过程中进行自我评分，以识别和减少可能出现的幻觉。Self-RAG 的流程从问题判断开始，根据需要选择是否进行检索；对检索到的文档进行评估和排序，判断检索到的文档是否能支撑生成内容；最后送入语言模型进行生成，并对生成的内容进行评估，判断是否结束响应或者重新生成。其重点在于通过反思评分来优化检索和生成过程，提高整体性能。

### Adaptive-RAG

Adaptive-RAG（Adaptive Retrieval-Augmented Generation）是一种更加灵活和动态的 RAG 变体，根据查询的复杂度自适应地选择最合适的检索策略。它使用一个小型的语言模型作为分类器来预测查询的复杂度，并根据复杂度级别选择相应的检索策略（如迭代式、单步式或无检索方法），核心思想是通过动态调整来平衡处理简单查询和复杂查询时的效率和准确性。

论文中作者指出两个观察到的现象：

- 用户大多数的问题都是简单的问题，少数情况下才是需要多跳推理的复杂问题。
- 简单的问题使用复杂的 RAG 模型存在开销的浪费，而复杂的问题又无法用简单的 RAG 模型来解决。

实现细节——将 user query 的难度分为三种，并给出对应的解决策略：

- Non Retrieval for QA：对于 Straightforward Query，不需要经过检索，直接由 LLM 回答。
- Single-step Approach for QA：对于 Simple Query，只需要经过一轮检索即可获取支持 LLM 回复的文档。
- Multi-step Approach for QA：对于 Complex Query，需要多步、复杂的检索才能得到答案。

分类器的训练：训练一个小语言模型作为 classifier（原论文使用 T5-large 再训练），输入是 user query，输出是 query 的难度。由于没有现成的 query-complexity pairs 数据集，论文借助已有的 QA 数据集来构建标注：

1. Generate silver data from predicted outcomes of models：给定一个 query，先让 LLM 直接回答，如果回答正确则将 query 标记为最简单等级；如果 LLM 经过一轮检索后生成正确答案，标记为中间等级；如果经过多轮检索后才生成正确答案，标记为最难等级。
2. Utilize inductive bias in datasets：经过第一个过程仍无法标记的 query（三种 RAG 模型都没有生成正确答案），利用 benchmark 数据集的偏向性来标注——偏 single-hop 的数据集直接标为中间等级，偏 multi-hop 的标为最难等级。

补充讨论：多次查询确实可能是多余的甚至反效果的，按难度自适应选择策略在处理性能和最终效果上都有收益；但难度分类本身效果较差，因为难度与大模型、知识库支持、问题领域之类的差异较大。

## CoT、ToT 和 ReAct

### CoT（Chain of Thought，思维链）与自我一致性

CoT 作为 Prompt 高级技巧的一部分，可以显著提高大模型在推理方面的能力，尤其是解决数学等具有逻辑性的问题。区别于传统 Prompt 从输入直接到输出的映射（input → output），CoT 完成了从输入到思维链再到输出的映射（input → reasoning chain → output）。

例如问题是"纽约到洛杉矶的距离是多少？"，模型可能首先检索纽约和洛杉矶的坐标，然后计算两点之间的距离，最后给出答案。模型不仅提供了答案，还展示了其推理过程，增强了答案的可信度。

使用 CoT：现在大模型本身就应该具备这个能力，只需要在 Prompt 中添加一句 "Let's think step by step" 就能让大模型在推理时显式生成中间推理步骤，并不需要自己写代码去实现 CoT。

Self-consistency with CoT（CoT 的自我一致性）：一种 CoT 在实际应用中的方案，生成多条推理路径并投票选择最优答案。它是 CoT 的后续方法：对同一个 Prompt 重复调用多次（如 5 次），然后取其中多数结果作为最终答案。

### ToT（Tree of Thought，思维树）

CoT 存在的缺陷：

- 对于局部，没有探索一个思考过程下的不同延续——树的分支。
- 对于全局，没有利用任何类型的规划、前瞻以及回溯去帮助评估不同抉择——而启发式的探索正是人类解决问题的特性。

针对以上缺陷，ToT 将推理过程扩展为树形结构，允许探索多分支路径并进行回溯，模拟人类深度思考。从原始输入开始，在思维链的每一步采样多个分支，逐渐形成树状结构，对每一个分支进行评估，然后在合适的分支上进行搜索。

在思维树中：

- 根节点：通常代表问题或决策的起点，即需要解决的核心问题。
- 分支：从根节点开始，每个分支代表一个可能的思考方向或解决方案。分支可以进一步细分，形成更详细的子分支。
- 叶节点：树的末端，代表思考过程的最终结果或结论。

通过构建思维树，可以系统地探索所有可能的思考方向、评估和比较不同分支的结果做出更合理的决策、通过可视化思考过程增强对问题的理解。目前 ToT 还没有得到特别好的效果，可能是在构建当中还有不合理的定义或者解析问题不精准的存在。

### ReAct

CoT、ToT 都是作用在大模型本身的内在推理（Reason）过程上，而 ReAct 则结合外部工具共同实现最终的目标（Reason + Action）。

ReAct 是一种结合了推理和行动的框架，主要用于增强 AI 系统在复杂环境中的决策能力和执行效率。核心思想是通过实时检索相关信息和执行基于这些信息的行动，来辅助 AI 系统进行更准确的推理和决策。AI 系统不仅依赖于其预训练的知识，还会在遇到新情况时主动检索外部信息（如数据库、网络资源等），并将这些信息整合到其决策过程中。这一过程是"思考"和"行动"之间的循环：

- 思考（Reasoning）：AI 系统基于当前状态和目标，进行推理和规划，确定下一步需要采取的行动或需要检索的信息。
- 行动（Acting）：根据推理结果，AI 系统执行相应的行动，如检索信息、执行任务等。
- 反馈：AI 系统根据行动的结果，更新其状态和知识，然后再次进入思考阶段，形成一个闭环。

ReAct 框架的优势在于使 AI 系统能够适应不断变化的环境，处理之前未见过的情况，而不仅仅依赖于预训练数据。

三者总结：CoT 目前已经集成进了大模型内部，通过在 Prompt 中加入提示词（Let's think step by step）即可唤醒大模型的 CoT 思考能力；ToT 是在思维链的基础上，每一步不再只有一个结果，而是采样多个分支综合评估；ReAct 则是从推理过程结合外部工具共同实现最终目标——ReAct 不止在推理，还在利用外部工具实现目标（Reason + Action），而 CoT、ToT 只是 Reason。

## Chunking 分块策略

### 文本切割的主要观点

几乎所有的大模型或嵌入模型，输入长度都是受限的，因此需要将文档进行分块。通过分块可以确保嵌入的内容尽可能少地包含噪音，同时嵌入内容和用户查询之间具有更高的语义相关性。主要观点：

- 文本切割是优化语言模型应用性能的关键步骤，切割策略应根据数据类型和语言模型任务的性质来定制。
- 传统的基于物理位置的切割方法（如字符级切割和递归字符切割）虽简单，但可能无法有效地组织语义相关的信息。
- 语义切割以及基于文本结构的切割是更高级的策略，通过分析文本的语义关系或文档结构来确定切割边界，从而保持语义完整性并提高检索效果。
- 使用多向量索引可以提供更丰富的文本表示。多向量索引是一种将文本数据以多个向量的形式进行表示和索引的技术，这些向量能够共同反映文本的多维度特征和信息。
- 工具和技术的选择应基于对数据的深入理解以及最终任务的具体需求。

### 固定大小分块与递归字符切割

固定大小分块（Fixed-size chunking）：字符级切割，简单粗暴地按字符数量固定切割文本。这是最常见也是最直接的分块策略，文档被分割成固定大小的分块（chunk_size），分块之间可以保留一些重叠（chunk_overlap），以确保不会出现语义相关的内容被不自然地拆分的情况。在大多数情况下固定大小分块都是最佳选择：既廉价又简单易用，而且不需要使用任何自然语言处理库。

分块大小是需要深思熟虑的参数，取决于所使用的嵌入模型的 token 容量。比如基于 BERT 的 sentence-transformer 最多只能处理 512 个 token，而 OpenAI 的 ada-002 能够处理 8191 个；另外也需要权衡大模型的 token 限制，因为分块大小直接决定了加载到大模型上下文窗口中的信息量。

递归字符切割：考虑文本的物理结构（如换行符、段落等），逐步递归切割。它接受一组分隔符（比如 `["\n\n", "\n", " ", ""]`），首先使用第一个分隔符对文本进行分块，如果分块后长度仍超出分块大小，则使用第二个，以此类推，通过递归迭代直到达到所需的块大小。

注意：大模型的上下文限制是 token 数量而不是文本长度，因此将文本分成块时，建议计算分块的 token 数量（比如使用 OpenAI 的 tiktoken 库）。

### 句子拆分与特定格式分块

句子拆分（Sentence splitting）：很多模型都针对句子级内容的嵌入进行了优化，如果能将文本按句子拆分，可以得到很好的嵌入效果。常见方法：

- 直接按英文句号（.）、中文句号（。）或换行符等进行分割：快速简单，但不会考虑所有可能的边缘情况，可能会破坏句子的完整性。
- NLTK：流行的自然语言工具包，提供句子分词器（sentence tokenizer），可以将文本分割成句子。
- spaCy：另一个强大的 NLP 库，提供复杂的句子分割功能，可以高效地将文本分割成单独的句子，在生成的块中更好地保留上下文。

特定格式分块（Specialized chunking）：很多文本文件具有特定的结构化内容（Markdown、LaTeX、HTML 或各种源码文件等），可使用专门的分块方法：

- Markdown：LangChain 的 MarkdownHeaderTextSplitter 通过 Markdown 的标题来组织分组，然后再在特定标题组中创建分块。
- HTML：LangChain 的 HTMLHeaderTextSplitter 根据标题来实现 HTML 的分块，HTMLSectionSplitter 能够在元素级别上分割文本，基于指定的标签和字体大小进行分割，将具有相同元数据的元素组合在一起。
- LaTeX：通过解析 LaTeX 可以创建符合内容逻辑组织的块（例如章节、子章节和方程式），LangChain 的 LatexTextSplitter 实现了 LaTeX 格式的分块。
- JSON：需要考虑嵌套 JSON 对象的完整性，通常按照深度优先的方式遍历 JSON 对象并构建出较小的 JSON 块，参考 LangChain 的 RecursiveJsonSplitter。
- PDF：使用 pypdf 按 page 逐页解析；使用 pdfplumber 逐页解析（但文本在分栏时存在混淆，解析不完全）；使用 PDFMiner 将整个文档解析成一个完整的文本；或使用非结构化工具 Unstructured。

### 语义分块与智能切割

语义分块（Semantic chunking）：利用语言模型的嵌入向量来分析文本的意义和上下文，以确定切割点。这是一种实验性的分块技术，最初由 Greg Kamradt 提出（他在 The 5 Levels Of Text Splitting For Retrieval 中将分块技术划分为 5 个等级，语义分块是第 4 级）。基本原理：

1. 首先将文本划分成一个个句子，并计算第一个句子的向量。
2. 接着计算第二个句子的向量，并和第一个句子进行比较，得到相似度；再计算第三个句子的向量与第二个句子比较，以此类推。
3. 当句子之间的相似度高于某个阈值时，说明这里的话题可能存在转折，可以将这个地方作为分块的临界点。

LangChain 的 SemanticChunker 和 LlamaIndex 的 SemanticSplitterNodeParser 都实现了语义分块。

智能切割：创建一个 Agent，使用 Agent 来决定如何切割文本，以更智能地组织信息。

## 句子窗口检索

普通 RAG 检索的流程：先将文档切片成大小相同的块 → 将切片后的块进行 Embedding 并保存到向量数据库 → 根据问题检索出 Embedding 最相似的 K 个文档块 → 将问题和检索结果一起交给 LLM 生成答案。普通 RAG 检索的问题是：如果文档切片比较大，检索结果可能会包含很多无关信息，从而导致 LLM 生成的结果不准确。

句子窗口检索的流程：

- 和普通 RAG 检索相比，句子窗口检索的文档切片单位更小，通常以句子为单位。
- 在检索时，除了检索到匹配度最高的句子，还将该句子周围的上下文也作为检索结果一起提交给 LLM。

原理：在文档切分时将文档以句子为单位进行切分——一句话相比一段话来说，语义可能更接近用户的问题——同时进行 Embedding 并保存数据库。在检索时，通过问题检索到相关的句子，但并不只是将检索到的句子作为检索结果，而是将该句子前面和后面的句子一起作为检索结果放到一个窗口中（窗口包含的句子数量可通过参数设置），最后将检索结果一起提交给 LLM 生成答案。

这与父文档检索的思想一致：通过检索更小的块获得更好的搜索质量，通过扩大上下文范围获取更好的推理结果。句子窗口检索让检索内容更加准确，同时上下文窗口又能保证检索结果的丰富性。

## Embedding 模型的发展

有很多嵌入模型可供选择，比如 BAAI 的 bge-large、微软的 multilingual-e5-large、OpenAI 的 text-embedding-3-large 等，可以在 MTEB 排行榜上了解最新的模型更新情况。

词嵌入技术经历了一个从静态到动态的发展过程：静态嵌入为每个单词使用单一向量，而动态嵌入根据单词的上下文进行调整，可以捕获上下文理解。排行榜上排名靠前的基本上都是动态嵌入模型。

关于嵌入模型的优化，通常围绕嵌入模型的微调展开，将嵌入模型定制为特定领域的上下文，特别是对于术语不断演化或罕见的领域。

值得一提的是，嵌入不仅仅限于文本，还可以创建图像或音频的嵌入，并将其与文本嵌入进行比较，这个概念适用于强大的图像或音频搜索、分类、描述等系统。

## RAG 工程实践要点

### 上下文压缩与效果评估

上下文压缩可以在多个阶段进行：

- 问题优化阶段：将聊天历史压缩成最终问题以便检索。
- 检索阶段：不立即按原样返回检索到的文档，而是对查询到的上下文进行压缩。
- 答案生成阶段：对生成的答案进行压缩。

RAG 效果评估的常用工具：RAGAS（可结合 LangSmith 评估 RAG pipeline）和 TruLens，它们从忠实度、答案相关性、上下文相关性等维度对 RAG 系统进行量化评估。

### 敏感信息处理与引用来源

敏感信息处理：检索的文档中可能含有如用户名、身份证、手机号等敏感信息，这类信息统称为 PII（Personal Identifiable Information，个人可识别信息）。如果将这类信息丢给大模型生成回复，可能存在安全风险，所以需要在后处理步骤中将 PII 信息删除。LlamaIndex 提供了两种方式删除 PII 信息：使用大模型（PIINodePostprocessor）和使用专用的 NER 模型（NERPIINodePostprocessor）。

引用来源：一个基于 RAG 的应用不仅要提供答案，还要提供答案的引用来源。这样做有两个好处：用户可以打开引用来源对大模型的回复进行验证；方便用户对特定主题进行进一步的深入研究。实现上可以在 Prompt 中要求大模型在生成内容时使用 [N] 格式表示来源，然后在客户端解析它并将其呈现为超链接（Perplexity 的 Prompt 即采用此方式）。

## 开发 RAG 系统面临的 12 个问题

### 七个常见失败点

论文《Seven Failure Points When Engineering a Retrieval Augmented Generation System》中提出了 RAG 系统中可能面临的 7 个问题：

1. 缺失内容（Missing Content）：用户的问题无法从文档库中检索到时，可能导致大模型的幻觉现象。理想情况下 RAG 系统可以简单回复"抱歉，我不知道"；然而如果用户问题能检索到文档，但文档内容和用户问题无关时，大模型还是可能被误导。
2. 错过超出排名范围的文档（Missed Top Ranked）：由于大模型的上下文长度限制，从文档库中检索时一般只返回排名靠前的 K 个段落，如果问题答案所在的段落超出了排名范围，就会出现问题。
3. 不在上下文中（Not In Context）：包含答案的文档已经成功检索出来，但却没有包含在大模型所使用的上下文中。当从数据库中检索到多个文档，并且使用合并过程提取答案时，就会出现这种情况。
4. 未能提取（Not Extracted）：答案在提供的上下文中，但是大模型未能准确地提取出来，这通常发生在上下文中存在过多的噪音或冲突信息时。
5. 错误的格式（Wrong Format）：问题要求以特定格式提取信息（例如表格或列表），然而大模型忽略了这个指示。
6. 不正确的具体性（Incorrect Specificity）：尽管大模型正常回答了用户的提问，但不够具体或者过于具体，都不能满足用户的需求。也可能发生在用户不确定如何提问或提问过于笼统时。
7. 不完整的回答（Incomplete Answers）：例如问题"文件 A、B、C 包含哪些关键点？"，直接使用这个问题检索得到的可能只是每个文件的部分信息，导致大模型的回答不完整。更有效的方法是分别针对每个文件提出问题，以确保全面覆盖。

### 另外五个工程问题

Wenqi Glantz 在博客《12 RAG Pain Points and Proposed Solutions》中又扩充了另 5 个问题：

8. 数据摄入的可扩展性问题（Data Ingestion Scalability）：当数据规模增大时，系统可能面临数据摄入时间过长、系统过载、数据质量下降以及可用性受限等问题，可能导致性能瓶颈甚至系统故障。
9. 结构化数据的问答（Structured Data QA）：根据用户的问题准确检索出所需的结构化数据是一项挑战，尤其是当用户的问题比较复杂或模糊时。文本到 SQL 的转换不够灵活，当前大模型在处理这类任务上仍存在局限性。
10. 复杂的 PDF（Complex PDFs）：复杂的 PDF 文档中可能包含表格、图片等嵌入内容，对这种文档进行问答时，传统的检索方法往往无法达到很好的效果，需要更高效的方法来处理复杂的 PDF 数据提取需求。
11. 备用模型（Fallback Models）：使用单一大模型时可能会遇到问题，比如 OpenAI 模型的访问频率限制错误，这时需要一个或多个模型作为备用，以防主模型出现故障。
12. 大语言模型的安全性（LLM Security）：如何有效地防止恶意输入、确保输出安全、保护敏感信息不被泄露等，都是需要面对的重要挑战。
